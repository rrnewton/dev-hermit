# OSS programs with known nondeterminism — candidates for a `hermit --strict` demo

Goal: real/idiomatic OSS programs whose **output varies run-to-run natively** and becomes **stable under `hermit --strict`**, demonstrating hermit's value. Date: 2026-07-22.

## Feasibility baseline (hermit's actual capabilities, from the repo)

- **Shared futexes now work** — PR #53 (merged) + `experiments/shared-futex-verify_20260722` verified **Node.js (Worker + SharedArrayBuffer/Atomics.wait) 3/3** and **OpenJDK (8-thread latch + AtomicInteger) 3/3** launch and pass under deterministic hermit. The old `EOPNOTSUPP` blocker for JVM/Node is gone.
- hermit `--strict` **sequentializes threads** and determinizes **scheduling, virtual time, and RNG/`getrandom`/`/dev/urandom`** (PR #59 "Unify deterministic getrandom and random-device handling").
- Python and Java already appear in hermit's arbitrary-binary matrix; python workloads run in the mode matrix.
- Caveats: heavy runtimes (JVM/Node) run **much slower** under sequentialized hermit; on **AMD hosts CPUID faulting is unavailable** so `RDRAND`/`RDTSC`-based nondeterminism can't be trapped (relevant only if a program reads those directly — most use `getrandom`/`clock_gettime`, which hermit does virtualize).
- **Already covered by existing hermit workloads** (don't duplicate): `mem_race`, `hello_race`, `cas_sequence`, `thread_random`, `print_race`, `futex_*`. The candidates below are *real programs / idiomatic patterns* beyond those.

Ranking = (demo value) × (likelihood hermit stabilizes it, given its real mechanisms) ÷ (effort to run). "Fix likelihood" reflects whether the nondeterminism source is one hermit actually controls (scheduling/time/RNG) vs. something it may not (raw hardware entropy on AMD).

## Ranked candidates

| # | Program / pattern | Category | Nondeterminism (source) | How `--strict` fixes it | Effort | Native nondet likelihood | Hermit-fix likelihood |
|---|---|---|---|---|---|---|---|
| 1 | **Python `set`/`dict` of strings iteration order** — `python3 -c "print(list({'apple','banana','cherry','date','fig'}))"` (or a small script printing a str-keyed dict) | Runtime/hashing | Order changes every run due to **string hash randomization** seeded from `getrandom`/`PYTHONHASHSEED` (bugs.python.org #40325, #27706) | hermit determinizes `getrandom`/urandom (#59) → the hash seed is fixed → iteration order is stable | **Very low** (interpreter already runs; one-liner) | **High** (guaranteed for str sets across runs unless `PYTHONHASHSEED` is pinned) | **High** — source is exactly what hermit virtualizes |
| 2 | **JVM thread-interleaving demo** — small Java program: N `Thread`s each printing tagged lines, or incrementing a **non-atomic** shared field, then print order/total | JVM / concurrency | Interleaved line order and lost-update totals vary with the OS thread schedule | `--strict` sequentializes threads into one deterministic schedule → one fixed order/total | **Medium** (JDK heavy but **proven to launch**; extend existing `experiments/.../Threaded.java` to use a non-atomic counter or interleaved prints) | **High** (print interleaving is very reliably nondeterministic) | **High** — thread scheduling is hermit's core guarantee |

| 5 | **Node.js async completion order** — fire N `fs.readFile` / `setTimeout(jitter)` and log completion order, or Worker threads racing on a `SharedArrayBuffer` | Node.js | Callback/promise resolution order varies with I/O and timer timing | hermit determinizes time + I/O completion + thread scheduling → fixed completion order | **Medium** (Node heavy but **proven to launch**; SharedArrayBuffer path already verified) | **Medium–High** (timer/IO jitter reliably reorders) | **Medium–High** (hermit controls time+IO+threads; microtask-vs-macrotask determinism worth verifying) |
| 6 | **JVM `HashSet`/`HashMap` iteration order via default identity `hashCode()`** — insert objects with no `hashCode` override, print iteration order | JVM / hashing | HotSpot's default identity hashCode is a **per-thread PRNG** seeded from `os::random()` → iteration order varies run-to-run (SitePoint "How Does the Default hashCode() Work?") | hermit determinizes the RNG/time source the seed comes from → stable identity hashcodes → fixed iteration order | **Medium** (JDK) | **High** (well-documented run-to-run variation) | **Medium** — depends on the JVM's seed path hitting a hermit-virtualized syscall; **verify empirically** |

| 8 | **Redis single-instance time/random commands** — script issuing `TIME`, `RANDOMKEY`, `SRANDMEMBER`, `SPOP`, TTL/expiry, `SCAN` and capturing replies | Database | Replies vary via wall-clock, PRNG, and hash-bucket cursor order; expiry depends on real time | Virtualized time+RNG → stable replies | **Medium–High** (build/run `redis-server` + drive a client; but Redis core is single-threaded — good fit) | **Medium** | **Medium** (time/rand fixable; `SCAN` bucket order depends on internal hashing + dict-rehash timing) |

## Recommendation (top 3 to build first)

1. **Python string-set ordering (#1)** — highest ROI: near-zero effort, guaranteed native nondeterminism, and the source (`getrandom`) is precisely what hermit determinizes. Ideal first "wow" demo.
2. **JVM thread-interleaving demo (#2)** — the flagship "hermit tames a real managed runtime" story; the JVM is already proven to launch under hermit, and only needs the existing `Threaded.java` asset changed to a non-atomic/interleaved variant so output actually varies natively.
3. **FP parallel reduction (#4)** — the most *credible-to-scientists* reproducibility story (bit-identical numerics), low effort, and squarely fixed by hermit's deterministic thread ordering.

Node.js (#5) and identity-hashCode (#6) are strong second-wave demos but carry empirical uncertainty (event-loop ordering; JVM seed path) worth a quick spike before committing.

## Suggested validation method (per candidate)

Run the program natively ~5× capturing stdout hashes (expect varied hashes), then `hermit --strict run -- <prog>` ~5× (expect one identical hash). This is the same varied-vs-stable oracle used in `experiments/shared-futex-verify_20260722`.
